# 批量成员导入设计

## 背景

当前「成员」页只支持管理员一次添加一个成员姓名到白名单。人数较多时，逐个添加效率低，也容易漏填。现有成员流程是：管理员先维护白名单，成员再使用姓名自行注册；白名单只控制谁可以注册，不直接生成账号和密码。

## 目标

本次新增“批量添加成员”能力，用于把多个姓名一次性加入白名单。

- 管理员可以在「成员」页粘贴名单。
- 推荐格式是一行一个姓名。
- 兼容 `、`、`,`、`;`、`；` 作为分隔符。
- 自动忽略空项。
- 自动识别并跳过本次输入中的重复姓名。
- 自动跳过已经存在于白名单中的姓名。
- 返回清晰结果：新增人数、跳过人数，以及必要的明细。

## 非目标

- 不直接创建用户账号。
- 不生成或设置成员密码。
- 不修改已注册用户。
- 不修改任何原始打卡记录。
- 不引入 CSV 或 Excel 文件上传。
- 不把空格作为主要分隔符，避免误拆姓名。

## 交互设计

在「成员」页当前单个姓名输入区旁增加「批量添加」按钮。

点击后打开弹窗：

- 标题：批量添加成员
- 输入区：多行文本框
- 占位提示：一行一个姓名，也可用顿号、逗号或分号分隔
- 操作按钮：取消、确认添加

提交成功后：

- 刷新白名单列表。
- 弹出结果提示，例如：`已添加 12 人，跳过 3 人`。
- 如果全部都是重复或已存在，则不报错，提示没有新增成员。

## 后端设计

新增管理员接口：

```http
POST /api/whitelist/bulk
Content-Type: application/json
```

请求体：

```json
{
  "names": ["张三", "李四", "王五"]
}
```

服务端处理规则：

- 只有管理员可调用。
- 每个姓名先做 `trim`。
- 空姓名不插入，计入 `invalid`。
- 本次请求内重复的姓名只插入一次，重复项计入 `skippedDuplicate`。
- 数据库中已存在的姓名不插入，计入 `skippedExisting`。
- 新增姓名写入 `whitelist`，`used=0`。

响应示例：

```json
{
  "ok": true,
  "added": ["张三", "李四"],
  "skippedExisting": ["王五"],
  "skippedDuplicate": ["张三"],
  "invalid": [],
  "counts": {
    "added": 2,
    "skipped": 2,
    "invalid": 0
  }
}
```

## 前端设计

前端负责把文本框内容解析成姓名数组，再调用批量接口。

解析规则：

- 换行是主要分隔符。
- 同时兼容 `、`、`,`、`;`、`；`。
- 不使用普通空格切分。
- 每项提交前做 `trim`。
- 空项不提交；如果绕过前端直接提交空项，后端会计入 `invalid`。

前端仍以服务端返回为准，不在本地判断“已存在”，避免前端列表过期造成误判。

## 错误处理

- 未登录或非管理员：沿用现有接口错误提示。
- 请求为空：前端提示“请输入成员姓名”。
- 服务端收到空列表：返回 400 和明确错误。
- 数据库写入失败：返回现有错误结构，前端 toast 显示失败原因。

## 文档更新

更新管理员使用指南的「成员管理」部分：

- 说明可单个添加，也可批量添加。
- 批量格式推荐一行一个姓名。
- 说明重复姓名会自动跳过。
- 明确批量添加只是加入白名单，成员仍需自行注册。

## 测试策略

后端测试：

- 批量新增多个姓名。
- 跳过数据库中已存在的姓名。
- 跳过本次输入中的重复姓名。
- 忽略空白姓名。
- 非管理员不能调用批量接口。

前端测试：

- 文本解析支持换行、顿号、逗号和分号。
- 普通空格不会拆分姓名。
- 提交成功后刷新白名单并显示汇总提示。
